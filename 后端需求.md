# 期刊平台后端需求文档

## 一、 角色与权限核心定义

系统包含多种编辑及用户角色，角色获取必须走审批流程（申请 / 提名 → 审核 → 任命 → 激活），不可自由注册（除普通作者/用户外）。

- **admin**：系统管理员，最高权限，负责系统配置、用户管理、数据备份等。
- **editor**：正式编辑，核心功能是终审判定、全流程统筹、团队权限分配，拥有最高权限，负责稿件全流程最终决策。所有编辑角色、权限、流程均由编辑委员会（Editorial Board）统一管理。一个编委账号即对应一个 Editor 角色。
- **associate_editor**：副编辑，核心功能是稿件初筛、推荐审稿人、审核评审意见。无最终决策权，仅提供专业领域参考建议。
- **ea_ae**：编辑助理/顾问编辑，辅助操作为主。负责流程文书处理、进度跟进、作者基础沟通；学术质量把控、争议稿件专业咨询。无评审与决策权限，不参与日常流程执行。
- **reviewer**：审稿人，核心功能是根据编辑推荐进行评审，评估稿件质量、符合期刊要求。
- **author**：作者，核心功能是提交稿件、管理个人信息、与编辑沟通。
- **user**：普通用户，核心功能是浏览期刊内容、注册账号、登录系统。

级别admin > editor > associate_editor > ea_ae > reviewer > author > user
高级别账号可向下登录各个站，并使用相关功能

## 二、 核心状态与阶段枚举

前后端交互需统一使用小写英文 `snake_case` 状态值。

### 1. 稿件流转阶段 (reviewStage / record.stage)
| 阶段类型 | 英文状态值 | 说明 |
|---------|------------|------|
| 初审 | `initial_review` | 稿件提交后的初步筛选 |
| 复审/同行评审 | `peer_review` | 分配给审稿人进行评审 |
| 终审 | `final_decision` | 编辑最终决策阶段 |

### 2. 稿件全局状态 (status)
| 流程节点 | 英文状态值 | 说明 |
|----------|------------|------|
| **初审阶段** | `pending_initial_review` | 待初审 |
| | `under_initial_review` | 初审中 |
| | `initial_review_passed` | 初审通过 |
| | `initial_review_revision` | 初审退回修改 |
| | `initial_review_rejected` | 初审拒稿 |
| **同行评审** | `pending_peer_review` | 待同行评审 |
| | `under_peer_review` | 评审中 |
| | `review_completed` | 评审完成 |
| | `revision_required` | 需要修稿 |
| | `revision_submitted` | 修订稿已提交 |
| **终审阶段** | `pending_final_decision` | 待终审 |
| | `under_final_decision` | 终审中 |
| | `final_decision_accepted` | 终审通过 |
| | `final_decision_revision` | 终审修改后录用 |
| | `final_decision_rejected` | 终审拒稿 |
| **录用出版** | `pending_acceptance_confirmation`| 待录用确认 |
| | `pending_copyright` | 待版权协议 |
| | `pending_proof` | 待校样确认 |
| | `pending_publication` | 待出版 |
| | `published` | 已出版 |
| **其他** | `withdrawn` | 已撤稿 |
| | `transfer_suggested` | 建议转投 |
| | `transferred` | 已转投 |

## 三、 API 接口定义摘要

系统接口遵循 RESTful 资源导向设计，按照角色通用（同一接口基于鉴权返回不同数据）和动作聚合的原则划分。

### 1. 认证与用户中心
- **认证 `/auth`**: `/login` (POST 登录), `/logout` (POST 登出), `/session/refresh` (POST 刷新), `/register` (POST 注册), `/password` (PUT 改密)
- **用户 `/users`**: `/me` (GET/PUT 个人信息), `/me/notifications` (GET 消息列表), `/me/notifications/{id}` (PUT 已读), `/` (GET 用户列表-管理), `/{id}` (PUT/DELETE 用户管理)

### 2. 稿件与文件中心
- **稿件 `/manuscripts`**: 
  - `GET / POST` : 稿件列表/创建
  - `GET / PUT / DELETE /{id}` : 详情/更新/删除
  - `POST /{id}/workflow` : **稿件流转核心接口** (支持动作: save, submit, withdraw, screen, assign, review, decide, revise, approve, publish)
  - `GET / POST / DELETE /{id}/files` : 附件管理
  - `GET / POST / PUT / DELETE /{id}/participants` : 参与者管理
  - `GET / POST /{id}/reviews` : 审稿记录/提交意见
  - `GET /{id}/history` : 操作历史
  - `GET / POST /{id}/letters` : 信函管理 (支持后端对接存储)
- **文件 `/files`**: `/` (POST 批量上传), `/{id}` (GET 下载, DELETE 删除, `/preview` 预览)

### 3. 审稿与编辑中心
- **审稿人 `/reviewers`**: `/` (GET 列表, POST 添加), `/{id}` (PUT/DELETE), `/me/tasks` (GET 任务), `/me/tasks/{id}` (PUT 处理)
- **编辑 `/editorial`**: `/dashboard` (GET 看板), `/board` (GET 编委列表, POST 添加), `/board/{id}` (PUT/DELETE)

### 4. 管理与公共中心
- **管理 `/admin`**: `/dashboard`, `/statistics/*` (系统与稿件统计), `/system` (配置), `/categories` (栏目管理), `/invitations` (邀请码), `/users` (用户管控)
- **公共 `/public`**: `/journals` (期刊列表), `/articles` (已发论文), `/search` (搜索), `/info` (期刊信息)

### 5. 通信辅助中心
- **通知 `/notifications`**: 邮件分发 (`/email`, `/email/batch`)，业务通知 (`/reviewer-invitation`, `/decision`, `/revision`, `/acceptance`)
- **信函 `/letters`**: GET/POST 列表/创建，`/{id}` PUT/GET 更新/详情
- **消息 `/messages`**: GET/POST 列表/发送，`/{id}/read` 标记已读，`/unread-count` 未读数

## 五、 基于后端技术栈的可参考实现

本项目后端采用现代异步架构构建，为保证高并发与高内聚，所有第三方组件及服务均已封装为全局单例类。在实现上述接口和逻辑时，请参考以下技术栈规范：

### 1. Web 框架：FastAPI
- **路由与鉴权**：使用 FastAPI 构建异步接口（`async def`），通过其强大的依赖注入系统（Dependency Injection）进行统一的权限校验（基于 JWT Token 的鉴权获取当前用户）。
- **参数校验**：利用 Pydantic 模型（Schemas）完成对请求体、查询参数的严格校验和序列化。

### 2. 关系型数据库：PostgreSQL (SQLAlchemy ORM)
- **管理方式**：通过 `backend/database/service/database_service.py` 封装的全局单例 `db_manager` 管理连接池与生命周期。
- **调用规范**：在接口中，通过 `async with db_manager.get_session() as session:` 获取 `AsyncSession`。
- **数据持久化**：所有的核心数据（如文献记录 Journal、用户信息 User、审计日志 AdminLog、邀请码 Invitation 等）均需定义为 SQLAlchemy 2.0 风格的 ORM 模型，实现表结构映射与关系查询。

### 3. 高频缓存与在线状态：Redis
- **管理方式**：通过 `backend/service/redis_service.py` 封装的全局单例 `redis_service`。
- **调用场景**：
  - **热点数据读取**：频繁访问且极少变更的系统配置、首页推荐文献等。
  - **在线用户与 Session 管理**：维护用户在线状态（如 `set_user_online`, `is_user_online`）、限流控制（Rate Limit）及图形/短信验证码缓存。

### 4. 模糊查询与全文检索：Elasticsearch
- **管理方式**：通过 `backend/service/elasticsearch_service.py` 封装的全局单例 `elasticsearch_service`。
- **调用场景**：
  - **文献倒排索引**：基于 IK 分词器存储文献的标题、摘要、作者、关键词等文本信息。
  - **高效检索**：前端调用 `/public/search` 等接口进行复杂多字段模糊匹配时，通过 `elasticsearch_service.search()` 获取命中的文献 ID 或用户 ID 列表，再（可选）通过 PostgreSQL 获取详细信息返回。

### 5. 异步消息与防并发队列：Kafka
- **管理方式**：通过 `backend/service/kafka_service.py` 封装的全局单例 `kafka_service`。
- **调用场景**：
  - **削峰填谷**：在投稿高峰期或批量发送审稿邀请、录用通知、系统邮件时，使用 `await kafka_service.send_message(topic, data)` 异步投递消息，避免阻塞主流程（Web 线程）。
  - **解耦操作**：业务模块将审计日志记录、状态变更事件等非核心依赖的逻辑发送至 Kafka，由后台消费者负责实际处理和落库。
